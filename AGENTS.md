# AGENTS_GENERAL.md

이 문서는 에이전트가 작업을 시작할 때 반드시 확인해야 하는 **공용 작업 지침서**입니다. 
모든 작업은 이 문서의 절차와 규칙을 기본값으로 따릅니다.

---

## 1. 전역 작업 원칙

### 의사결정 및 태도
- **근거 기반 판단:** 추측성 결론을 배제하고 코드, 로그, 문서 등 확실한 근거에 기반해 보수적으로 판단합니다. 근거가 부족하면 사용자에게 확인을 요청합니다.
- **최소 변경 원칙:** 요청받은 범위 내에서만 수정하며, 범위 밖의 개선 사항은 제안 후 승인을 얻어 별도 작업으로 분리합니다.
- **보안 준수:** API 키, 비밀번호, `.env` 파일 등 민감 정보는 절대로 출력하거나 외부로 노출하지 않습니다.

### 커뮤니케이션
- **언어:** 기본 응답 및 커밋 메시지는 **한국어**를 사용합니다.
- **코드 인용:** 코드 근거를 제시할 때는 파일 경로와 행(Line) 정보를 반드시 포함합니다.
- **보고 구조:** 작업 완료 시 `작업 결과`, `검증 내용`, `남은 리스크`를 구분하여 보고합니다.

## 2. 표준 작업 순서 (Workflow)

에이전트는 반드시 다음 단계를 준수하며 작업을 진행합니다.

1. **Issue 생성:** 작업 목표와 범위를 정의한 이슈를 먼저 생성합니다.
   - 생성 전 제목과 본문 초안을 사용자에게 공유하고 승인을 받습니다.
2. **Branch 생성:** 이슈 번호를 포함한 작업용 브랜치를 생성합니다. (예: `feat/#12-login-api`)
3. **작업 및 검증:** 코드를 수정하고, 로컬 테스트 및 문법 검사를 수행합니다.
4. **Commit:** 변경 사항을 논리적 단위로 나누어 커밋합니다.
5. **Pull Request (PR):** 기준 브랜치로의 병합을 위한 PR을 생성합니다.
   - PR 본문에 `Closes #이슈번호`를 명시하여 이슈와 자동 연결합니다.
   - 생성 전 초안 승인을 반드시 거칩니다.

## 3. 코드 품질 및 CI 규칙

### 작업 규모 제한 (Atomic Task)
- **PR 크기:** 전체 변경량(추가+삭제)은 **600줄 이하**, 변경 파일은 **10개 이하**를 권장합니다.
- **파일 크기:** 단일 파일당 변경량은 **150줄 이하**로 유지하여 리뷰 가독성을 높입니다.

### 검증 강제
- **Pre-commit:** 커밋 전 스타일 가이드 및 정적 분석 도구(`lint`, `format`) 검사를 반드시 통과해야 합니다.
- **문법 검사:** 언어별 표준 문법 검사(예: Python의 `py_compile`)를 수행하여 런타임 에러를 방지합니다.

## 4. 메타데이터 관리

이슈와 PR 생성 시 다음 항목을 필수적으로 설정하여 관리 효율을 높입니다.

- **Assignee:** 작업 담당자(기본값: 사용자 본인)를 지정합니다.
- **Labels:** 작업 유형(예: `feature`, `bug`, `documentation`)에 맞는 라벨을 부착합니다.
- **Milestone:** 현재 진행 중인 목표(Milestone)가 있는 경우, 작업을 연결합니다.
- **Project:** 프로젝트 보드(Project Card)를 운영 중인 경우 해당 보드에 연결합니다.
- **Hierarchy:** 대규모 작업은 상위(Parent) 이슈 아래 하위(Sub-issue) 이슈로 구조화합니다.

## 5. 문서 관리 원칙

- **동기화:** 코드 수정으로 인해 문서 내용이 변경되어야 하는 경우, 문서 업데이트를 동일 작업 범위에 포함합니다.
- **인덱스 갱신:** 새로운 문서 파일 추가 시, 상위 인덱스 문서에 해당 내용을 반드시 반영합니다.
